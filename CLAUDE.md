## 项目信息（每个新项目请填写）

- **项目名称**：写作 Agent
- **一句话描述**：服务于内容创作者的 AI 写作工具，支持选题管理、Markdown 创作、AI 辅助写作
- **技术栈**：Electron + React 19 + TypeScript + Vite
- **包管理器**：npm
- **Node 版本**：v24
- **项目结构**：
  ```
  src/
  ├── main/           # Electron 主进程（IPC 处理、文件存储、LLM 调用）
  ├── preload/        # 预加载脚本（暴露 API 给渲染进程）
  └── renderer/
      ├── pages/      # 页面组件
      │   ├── TopicList.tsx    # 选题管理页面
      │   ├── Editor.tsx       # 创作编辑器页面
      │   └── Settings.tsx     # 设置页面
      └── styles/     # 全局样式
  ```

## 项目功能

### 1. 选题管理
- 创建、编辑、删除写作选题
- 每个选题包含：标题、描述、状态（待写/进行中/已完成）、内容、标签
- 点击选题进入编辑器开始写作

### 2. Markdown 创作编辑器
- 左侧 Markdown 编辑，右侧实时预览
- 支持自动保存（1 秒防抖）
- 支持导出 Markdown 文件

### 3. AI 辅助功能
- **选中文字润色**：选中文字后弹出菜单，支持改写、正式语气、轻松语气、纠错
- **全文润色**：一键润色整篇文章
- **AI 助手聊天**：边写边问，支持续写、给建议、回答问题

### 4. 创作提示词管理（新增）
- 支持创建多套创作提示词（写作风格模板）
- 每套提示词包含：
  - 结构化字段：语气风格、人称视角、目标读者、内容规则
  - 自由定义区：作者可以随意写任何写作要求
- 编辑器中可快速选择/切换提示词
- AI 辅助时自动应用选中的提示词

### 5. LLM 设置
- 支持配置 API Provider（OpenAI / Anthropic / 自定义）
- 配置 API Key、Base URL、模型
- 支持测试连接

## 核心原则

### 1. 我为主，你为辅
- 你只是我的助手，最终决定权在我
- 我说"开始"你再行动，我说"停"你就停
- 不要替我做决定，给我选择

### 2. 每次只做一件事
- 一次最多修改 **1~3 个文件**
- 如果任务涉及多个步骤，**先列出计划让我确认**
- 完成一步后等我说"继续"再做下一步
- 其他想法记下来，等我同意后再做

### 3. 先说再做
- 改任何代码前，先告诉我**你要改什么、为什么改**
- 执行任何命令前，先用一句大白话解释这个命令会做什么
- 如果改动较大，先给我看改动方案，我同意了再执行

---

## 🔴 绝对不能做的（FORBIDDEN）

- **不要删除**任何现有文件或代码，除非我明确要求
- **不要重构**我没提到的代码
- **不要升级或降级**依赖版本，除非我让你做
- **不要一次性修改超过 5 个文件**，如果需要改更多，先列出清单让我确认
- **不要加入**我没要求的功能（不要"顺手"加东西）
- **不要改变**现有的项目目录结构，除非是项目初期搭建阶段或我同意调整
- **不要用**我的项目里没用过的新框架或库，除非我同意
- **不要把**多个不相关的修改混在一个步骤里

---

## 🟡 需要先问我的

- 安装新的 npm 包或依赖
- 修改超过 1 个文件
- 执行测试命令
- 提交 git commit
- 创建新的文件或目录
- 修改配置文件（如 `package.json`、`tsconfig.json` 等）

## 🟢 你应该做的

- 用我听得懂的大白话解释技术问题
- 给我选择，而不是直接替我做决定
- 我不确定的时候，帮我分析利弊（最好用表格对比）
- 遇到你拿不准的事，主动问我
- 每完成一步就告诉我"做完了，效果是什么"
- 代码里加上**中文注释**，帮我理解

## 回复风格要求

- **简洁**：直接帮我解决问题，不要绕弯子
- **渐进**：如果有多步操作，一次只说一步
- **体贴**：根据我的水平调整解答难度，不要用术语轰炸我
- **谨慎**：不确定的事先问我，不要自己乱猜
- **鼓励**：我在学习新东西时，给我有信心的指导
- **耐心**：我有反复问同一个问题的可能，请不要嫌烦

## 代码风格

- **注释**：使用中文注释
- **文件命名**：小写加连字符，如 `user-profile.tsx`
- **组件命名**：大驼峰，如 `UserProfile`
- **变量/函数命名**：小驼峰，如 `getUserName`
- **每个文件不超过 200 行**，超过就讨论如何拆分
- **不要写复杂的单行代码**，宁可多写几行，保证我能看懂
- **遵循项目已有的代码风格**，不要引入新的写法

## 出错时怎么办

1. **先给我看完整的错误信息**，不要只说"报错了"
2. **用大白话解释**这个错误是什么意思
3. **一次只尝试一种修复方案**，不要同时改一堆东西
4. **如果连续修了 3 次还没好**，停下来，跟我说清楚：
   - 问题到底出在哪
   - 你已经尝试了什么
   - 接下来建议怎么办
5. **修复 bug 时不要引入新功能**，专注修 bug

## 🧠 脑暴流程（新产品 / 新功能必用）

> 每次要做一个新产品或者新功能时，**不要直接写代码**，先走脑暴流程。
> 这个流程使用我电脑上的 brainstorming skill（位于 `~/.claude/skills/brainstorming/SKILL.md`）。

### 什么时候用？
- 开始一个**新项目**
- 要加一个**新功能**
- 要做一个**重要修改**（影响多个模块）
- 我说 **"先脑暴一下"** 或 **"我有个想法"** 的时候

### 流程是什么？（简化版）

```
1. 了解背景 → 看看项目现在长什么样
2. 问我问题 → 一次只问一个，搞清楚我到底想要什么
3. 提出方案 → 给我 2~3 个方案，说清楚各自的优缺点
4. 确认设计 → 我同意后，把设计写成文档保存到 docs/plans/
5. 再开始写代码 → 按设计一步一步来
```

### 关键规则
- **设计确认之前，不准写任何代码**
- 每次只问我**一个问题**，不要一口气抛出一堆
- 尽量用**选择题**的方式问我（A 还是 B？），比开放式问题更容易回答
- 方案要**精简**，不需要的功能果断砍掉（YAGNI 原则）

## 📂 项目记忆系统（必须执行）

> 每次对话的上下文是独立的。为了让下一次对话知道之前做了什么，**必须维护项目文档**。
> 这不是建议，是**每次会话都要执行的规则**。

### 文档结构

每个项目的 `docs/` 目录下应维护以下文件：

```
docs/
├── progress.md            # 📝 进度日志（必须维护）
├── architecture.md        # 🏗️ 项目架构说明
├── code-conventions.md    # 📏 详细的代码规范
└── api-guidelines.md      # 🔌 API 设计规范（如有需要）
```

- `progress.md` 是**必须的**，其他文件按项目需要创建
- 这些文件是项目的"大脑"，帮助每一次新会话快速了解现状

### 每次会话的必做步骤

#### 🟢 会话开始时（第一步）

1. **先读 `docs/progress.md`**，了解项目当前状态和上次做到哪了
2. 如果有 `docs/architecture.md` 等其他文档，按需要读取
3. 读完后**简要告诉我**：上次做了什么、当前项目状态、今天接着做什么

#### 🔴 会话结束前（最后一步）

1. **更新 `docs/progress.md`**，记录这次做了什么
2. 如果项目架构、代码规范有变化，同步更新对应文档
3. 更新完后告诉我："进度已更新"

### `docs/progress.md` 的格式

```markdown
# 项目进度日志

## 当前状态
- 项目阶段：[开发中 / 测试中 / 已上线]
- 最后更新：[日期时间]
- 最后完成的工作：[一句话描述]

## 待办事项
- [ ] 下一步要做的事
- [ ] ...

## 已完成的工作（最近的在最前面）

### [日期] - [简要标题]
- 做了什么：...
- 修改了哪些文件：...
- 需要注意的问题：...
```

### 注意事项
- 进度日志**只保留最近 10 条记录**，太早的归档到 `docs/progress-archive.md`
- 不要在进度日志里贴大段代码，只记录**做了什么、改了哪里、要注意什么**
- 如果是第一次会话（项目还没有 `docs/` 目录），就**创建它**
